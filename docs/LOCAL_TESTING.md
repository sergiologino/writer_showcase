# Локальный запуск для тестирования (Windows 11)

Инструкция для машины с установленными **PostgreSQL**, **IntelliJ IDEA**, **JDK 17** (или новее, совместимый с проектом), **Node.js** и **npm**. Пути и команды рассчитаны на корень репозитория **writer_showcase**.

---

## Краткий порядок действий

1. Создать в PostgreSQL базу и пользователя (или использовать уже существующие и поправить переменные).
2. Запустить бэкенд из IDEA (или из терминала).
3. В отдельном терминале установить зависимости фронта и выполнить `npm run dev`.
4. Открыть в браузере адрес фронта (по умолчанию **http://localhost:5173**).

---

## 1. База данных PostgreSQL

При первом старте бэкенд сам применит схему (**Flyway**). Нужна пустая база и пользователь с правами на неё.

Пример (выполнить в **psql** или через pgAdmin), если хотите совпасть с настройками по умолчанию в приложении:

```sql
CREATE USER publisher WITH PASSWORD 'publisher';
CREATE DATABASE publisher OWNER publisher;
```

Если имя базы, пользователь или пароль другие — укажите их в переменных `DATASOURCE_*` при запуске бэкенда (см. раздел ниже).

**Важно:** служба PostgreSQL должна быть запущена, порт (обычно **5432**) должен совпадать с тем, что в строке подключения.

## 2. Бэкенд (Spring Boot)

### Вариант А — IntelliJ IDEA

1. **File → Open** и выберите папку **`apps/backend`** (или весь монорепозиторий — тогда настройте рабочую папку явно, см. п. 4).
2. Дождитесь импорта Maven (`pom.xml`).
3. Найдите класс **`io.altacod.publisher.PublisherApplication`**.
4. **Run → Edit Configurations…** → добавьте конфигурацию **Spring Boot** (или **Application**):
   - **Main class:** `io.altacod.publisher.PublisherApplication`
   - **Working directory:** для монорепо укажите **`$ProjectFileDir$/apps/backend`** (или полный путь к `writer_showcase\apps\backend`), чтобы корректно подхватывались `src/main/resources`, каталог хранилища файлов и **файлы `.env` / `.env.local`**.
5. **Переменные окружения (удобный способ):** создайте в каталоге **`apps/backend`** файл **`.env.local`** (рядом с `pom.xml`). Шаблон — **`apps/backend/.env.example`**: скопируйте и раскомментируйте нужные строки. Формат: `KEY=value`, без пробелов вокруг `=`, одна переменная на строку. При старте приложение подставит их как настройки (профиль **`test`** для автотестов это **не** читает — тесты по-прежнему на встроенной БД). Опционально можно добавить общий **`.env`**, а в **`.env.local`** переопределить только своё; приоритет у **`.env.local`**.
6. **Если в IDEA всё же есть поле Env (или VM options):** его можно оставить пустым при использовании `.env.local`. В старых версиях Spring Boot-конфигурации поле может называться **Environment variables** или быть скрыто в разделе **«Модифицировать параметры» / Modify options** → **Environment variables** — это не обязательно, если используете файлы.
7. Запустите конфигурацию. В логе не должно быть ошибок подключения к БД и миграций.

Проверка: приложение слушает порт **8080** по умолчанию (если не меняли `SERVER_PORT`). Отдельной «приветственной» страницы может не быть; убедитесь по логу строкой вроде `Started PublisherApplication`.

### Вариант Б — терминал (без IDEA)

В PowerShell из папки `apps\backend`:

```powershell
.\mvnw.cmd spring-boot:run
```

Перед этим можно задать переменные в этой же сессии (`$env:DATASOURCE_URL = "..."`) или положить **`apps/backend/.env.local`** (см. выше) и запускать из каталога **`apps\backend`**, чтобы совпадал `user.dir`.

---

## 3. Фронтенд (Vite + React)

В PowerShell из корня репозитория:

```powershell
cd apps\web
npm install
npm run dev
```

По умолчанию dev-сервер: **http://localhost:5173** и запросы к API идут через **прокси Vite** на `http://localhost:8080` (отдельно ничего не настраивать, если бэкенд на 8080).

Если фронт и бэкенд на **разных адресах** (другой порт или другой хост без общего прокси), создайте в `apps\web` файл **`.env`** на основе **`.env.example`** и задайте **`VITE_API_BASE_URL`** (см. таблицу ниже).

---

## 4. Опционально: Redis

Для локальной проверки приложение **работает и без Redis** — очередь публикаций в соцсети держится в памяти до перезапуска бэкенда.

Если нужно, как в «боевом» режиме, общая очередь между перезапусками:

1. Установите и запустите **Redis** на **localhost:6379** (или поднимите контейнер из `docker-compose.yml` в корне репозитория — там есть сервис `redis`).
2. В переменных бэкенда включите:

   ```text
   PUBLISHER_REDIS_ENABLED=true;REDIS_HOST=localhost;REDIS_PORT=6379
   ```

3. При пустом пароле у Redis переменную **`REDIS_PASSWORD`** не задают.

---

## 5. Ссылки в сообщениях Telegram / ВК

Чтобы в автосообщениях в каналы подставлялась **правильная ссылка на пост** на вашем локальном фронте, задайте:

```text
PUBLIC_SITE_BASE_URL=http://localhost:5173
```

(без слэша в конце или со слэшем — оба варианта обычно обрабатываются; лучше без завершающего `/`).

## Переменные окружения: бэкенд

Ниже имена **как в коде** — их удобно копировать в Run Configuration IDEA или в систему. Значения с примером по умолчанию или по смыслу.

| Переменная | Зачем нужна | Пример / подсказка |
|------------|-------------|---------------------|
| `DATASOURCE_URL` | Подключение к PostgreSQL | `jdbc:postgresql://localhost:5432/publisher` |
| `DATASOURCE_USERNAME` | Пользователь БД | `publisher` |
| `DATASOURCE_PASSWORD` | Пароль БД | ваш пароль |
| `JWT_SECRET` | Подпись «сеансовых» токенов входа | **Длинная** случайная строка (десятки символов), не для продакшена из примеров в документах |
| `JWT_ACCESS_TTL_MINUTES` | Сколько минут живёт «короткий» токен | `60` |
| `JWT_REFRESH_TTL_DAYS` | Сколько дней можно обновлять сессию без пароля | `30` |
| `SERVER_PORT` | Порт HTTP бэкенда | `8080` |
| `CORS` (в `application.yml`) | Для локальной разработки по умолчанию разрешены **`http://localhost:*`** и **`http://127.0.0.1:*`** (любой порт Vite: 5173, 5174 и т.д.). Для продакшена задайте в профиле **`publisher.cors.allowed-origins`** или пустой список паттернов и точные origin — см. `DEPLOY_BACKEND.md` |
| `RATE_LIMIT_ENABLED` | Ограничение частоты запросов с одного IP | `true` / `false` — для спокойного ручного теста иногда отключают |
| `RATE_LIMIT_AUTH_PER_MINUTE` | Лимит запросов на вход/регистрацию в минуту | `40` |
| `RATE_LIMIT_API_PER_MINUTE` | Лимит остальных API-запросов в минуту | `400` |
| `STORAGE_LOCAL_ROOT` | Папка на диске для загруженных файлов | например `storage` относительно рабочей папки `apps\backend` или абсолютный путь |
| `STORAGE_MAX_UPLOAD_BYTES` | Максимальный размер одной загрузки в байтах | по умолчанию ~50 МБ |
| `MULTIPART_MAX_FILE_SIZE` | Лимит одной части multipart | по умолчанию 50MB |
| `MULTIPART_MAX_REQUEST_SIZE` | Лимит всего запроса с файлами | по умолчанию 55MB |
| `PUBLISHER_REDIS_ENABLED` | Использовать ли Redis для очереди публикаций в соцсети | `false` локально достаточно; `true` если Redis запущен |
| `REDIS_HOST` | Хост Redis | `localhost` |
| `REDIS_PORT` | Порт Redis | `6379` |
| `REDIS_PASSWORD` | Пароль Redis, если включён | часто пусто locally |
| `PUBLIC_SITE_BASE_URL` | Базовый адрес сайта для **ссылок в Telegram/ВК** | `http://localhost:5173` |
| `AI_INTEGRATION_BASE_URL` | Адрес сервиса ИИ (noteapp-ai-integration) | пусто — блок ИИ недоступен; иначе URL; **приоритетнее** `AI_INTEGRATION_URL`, если заданы оба |
| `AI_INTEGRATION_URL` | Дублирует смысл базового URL (как в других сервисах) | например прод: `https://sergiologino-zettelkastenapp-ai-integration-bce3.twc1.net` |
| `AI_INTEGRATION_API_KEY` | Ключ доступа к сервису ИИ | секрет, как выдаёт сервис |
| `INTEGRATION_AI_BASE_URL` | Дублирует смысл `AI_INTEGRATION_*` для совместимости | можно не задавать, если задан `AI_INTEGRATION_BASE_URL` |
| `INTEGRATION_AI_API_KEY` | То же для ключа | опционально |
| `AI_INTEGRATION_API_KEY_HEADER` | Заголовок с ключом | по умолчанию `X-API-Key` |
| `AI_INTEGRATION_PROCESS_PATH` | Путь метода обработки на сервисе ИИ | по умолчанию `/api/ai/process` |
| `AI_INTEGRATION_AVAILABLE_NETWORKS_PATH` | Путь `GET` списка сетей (тот же ключ, что и для `process`) | по умолчанию `/api/ai/networks/available` |
| `INTEGRATION_AI_AVAILABLE_NETWORKS_PATH` | Синоним пути списка сетей | опционально |
| `AI_INTEGRATION_CONNECT_TIMEOUT_MS` | Таймаут соединения с ИИ | `5000` |
| `AI_INTEGRATION_READ_TIMEOUT_MS` | Таймаут чтения ответа ИИ | `120000` |
| `ADMIN_EMAILS` | Список email с правами глобального админа (через запятую) | например `you@domain.com`; дополняет `is_admin` в БД — см. `docs/ai/AI_INTEGRATION.md` |
| `PUBLISHER_ADMIN_BOOTSTRAP_ENABLED` | Создать/пометить админа при старте (dev) | `false` — см. `docs/ai/AI_INTEGRATION.md` |
| `PUBLISHER_ADMIN_BOOTSTRAP_EMAIL` / `PASSWORD` / `DISPLAY_NAME` | Учётка bootstrap-админа | значения по умолчанию в коде, в проде не использовать без смены пароля |
| `PUBLISHER_CHANNEL_POLL_MS` | Как часто опрашивается очередь исходящих сообщений (мс) | по умолчанию `2000` |
| `PUBLISHER_CHANNEL_RETRY_POLL_MS` | Как часто проверяются отложенные повторы отправки (мс) | по умолчанию `45000` |
| `PUBLISHER_CHANNEL_MAX_ATTEMPTS` | Сколько раз система будет повторять неудачную отправку в канал | по умолчанию `8` |
| `PUBLISHER_CHANNEL_BACKOFF_BASE_SEC` | Базовая пауза между попытками (сек) | `45` |
| `PUBLISHER_CHANNEL_BACKOFF_MAX_SEC` | Потолок паузы (сек) | `7200` |
| `PUBLISHER_CHANNEL_BACKOFF_JITTER_PCT` | Разброс паузы, % | `25` |
| `PUBLISHER_CHANNEL_RETRY_BATCH` | Сколько отложенных заданий обрабатывать за один проход | `40` |

Для чисто ручного сценария «открыл сайт, прошёл сценарии» чаще всего хватает: **`DATASOURCE_*`**, **`JWT_SECRET`**, при проверке TG/VK и ссылок — **`PUBLIC_SITE_BASE_URL`**. CORS для localhost обычно **ничего не задают** — уже настроено в `application.yml`.

Сводно по **интеграции с нейросетями** (доп. переменные, smoke-проверки, bootstrap-админ): [`docs/ai/AI_INTEGRATION.md`](ai/AI_INTEGRATION.md).

## Переменные окружения: фронтенд

Файл **`apps/web/.env`** (не коммитится; образец — **`apps/web/.env.example`**). Префикс **`VITE_`** обязателен для переменных, видимых в браузере.

| Переменная | Зачем нужна | Пример / подсказка |
|------------|-------------|---------------------|
| `VITE_API_BASE_URL` | Полный адрес бэкенда, если **не** используете встроенный прокси dev-сервера | Пусто: запросы идут на тот же хост, что и фронт, через proxy на 8080. **`http://localhost:8080`** — если открываете собранный фронт отдельно и API на 8080 без прокси |

После изменения `.env` перезапустите **`npm run dev`**.

## Что проверить после запуска

1. Регистрация и вход.
2. Создание/редактирование поста, загрузка вложения.
3. Публичные страницы блога (без входа), если пост опубликован и виден всем.
4. Настройки каналов Telegram/ВК и появление сообщения в канале при **первой** публикации поста «для всех» — нужны реальные тестовые токен и чат/группа.

## Типичные проблемы

| Симптом | Что проверить |
|--------|----------------|
| Ошибка CORS в консоли браузера | Убедитесь, что открываете фронт с **`http://localhost`** или **`127.0.0.1`** (любой порт). Если фронт на **другом домене** — добавьте его в **`publisher.cors`** на бэкенде. |
| Бэкенд не стартует, ошибка БД | Запущен ли PostgreSQL, существует ли база, верны ли `DATASOURCE_*`. |
| 401 / вылетает сессия | Убедитесь, что **`JWT_SECRET`** не меняли между запусками при тех же пользователях в БД; при смене секрета перелогиньтесь или очистите данные теста. |
| Фронт «не достучался» до API | Бэкенд на 8080, фронт через `npm run dev`; если другой порт бэкенда — настройте прокси в `vite.config` или `VITE_API_BASE_URL`. |
