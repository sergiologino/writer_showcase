# Архитектурные решения

Формат: кратко **решение**, **контекст**, **последствия**. Устаревшие решения помечаются **Superseded** с ссылкой на замену.

---

## ADR-001 — Модульный монолит (backend)

- **Решение**: целевая форма backend — модульный монолит с доменными модулями, а не набор микросервисов на старте.
- **Причина**: единая модель, проще MVP и деплой; соответствует разделу 5 ТЗ (`altacod_publisher_spec.md`).
- **Последствия**: границы модулей и пакетов нужно держать явными с первых коммитов, чтобы при росте была возможность выделения сервисов без полной переписки.

---

## ADR-002 — Стек технологий

- **Решение**: использовать стек из разделов 4.1–4.2 ТЗ (Spring Boot 3, PostgreSQL, Redis, React, TypeScript и перечисленные библиотеки) как целевой для реализации, если не будет явного запроса на изменение.
- **Причина**: ТЗ задано как архитектурный blueprint.
- **Последствия**: отклонения фиксировать здесь новым ADR.

---

## ADR-003 — Монорепозиторий `apps/backend` + `apps/web`

- **Решение**: один репозиторий с двумя приложениями и общей сборкой через корневой `npm run build`.
- **Причина**: удобство MVP, один PR, общая версияность.
- **Последствия**: CI/деплой настраивать на два шага или два артефакта.

---

## ADR-004 — Java 17 вместо 21 на этапе bootstrap

- **Решение**: `target` сборки **17** под доступный JDK; ТЗ допускает 21+ как цель.
- **Причина**: на текущей среде разработки установлен JDK 17.
- **Последствия**: поднять `java.version` до 21 при появлении JDK 21 без смены остального стека.

---

## ADR-005 — Тесты БД: H2 + Flyway, изоляция по имени БД

- **Решение**: интеграционные тесты используют H2 в режиме PostgreSQL и те же Flyway-скрипты; уникальный JDBC URL (`random.uuid`), Hibernate диалект из драйвера (не форсировать PostgreSQLDialect в тестах).
- **Причина**: Docker/Testcontainers недоступен без демона; Hibernate с PostgreSQLDialect генерирует SQL, несовместимый с H2.
- **Последствия**: миграции держать совместимыми с H2 и PostgreSQL (типы времени — `timestamp with time zone`, без `TIMESTAMPTZ` в чистом виде для H2).

## ADR-006 — Публичный API с префиксом workspace slug

- **Решение**: публичные посты адресуются как `/api/public/w/{workspaceSlug}/posts...`, UI `/blog/{workspaceSlug}`.
- **Причина**: мультитенантность по workspace без отдельного домена на MVP.
- **Последствия**: при кастомных доменах на пост — отдельный ADR (маршрутизация и SEO).

---

## ADR-007 — Docker Compose для локальной БД

- **Решение**: в корне репозитория `docker-compose.yml` с сервисом **PostgreSQL 16** (порт `5432`, БД/пользователь `publisher`); удобные npm-скрипты `docker:up` / `docker:down` / `docker:logs` в корневом `package.json`.
- **Причина**: единообразное поднятие Postgres без ручной установки; совпадает с дефолтами `application.yml`.
- **Последствия**: приложения запускаются на хосте и подключаются к `localhost:5432`. Контейнеризация backend/web в compose — отдельный шаг при необходимости.

---

## ADR-008 — Целевой публичный URL автора (после MVP)

- **Решение (цель)**: у каждого автора публичный базовый путь вида `https://example.com/{author_nickname}/...` (ник вместо `workspace.slug` в URL).
- **Сейчас (MVP)**: публичные пути остаются на `workspace.slug` в блоге и API; переход на nickname — отдельная задача (ADR-008).
- **Последствия при внедрении**: поле/уникальность nickname, маршрутизация и обратная совместимость или редиректы с `/blog/{workspaceSlug}`.

## ADR-009 — Образ backend и профиль Compose `backend`

- **Решение**: в репозитории **`apps/backend/Dockerfile`** (multi-stage: Maven в образе, run на JRE 17 Alpine, непривилегированный пользователь). Сервис **`api`** в `docker-compose.yml` включён профилем **`backend`**, чтобы по умолчанию поднималась только PostgreSQL.
- **Причина**: деплой в Coolify и локальный запуск API в Docker без Windows `mvnw.cmd` внутри контейнера.
- **Последствия**: для сборки образа нужен запущенный Docker; переменные `JWT_SECRET` / `CORS_ORIGINS` для compose можно задать на хосте.

## ADR-010 — Опциональный `VITE_API_BASE_URL`

- **Решение**: при сборке фронта необязательная переменная **`VITE_API_BASE_URL`** префиксует запросы `apiFetch`; без неё сохраняются относительные `/api/...` (прокси Vite или единый nginx-origin).
- **Причина**: SPA и API на разных доменах без обязательного reverse proxy на стороне фронтенд-хоста.
- **Последствия**: на backend нужен корректный **`CORS_ORIGINS`** под origin SPA.
